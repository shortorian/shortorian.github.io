I live in Seattle so I’ve become increasingly familiar with the effect of wildfires in recent years. In the last few days, though, the air quality has been as bad as I’ve ever experienced. This map reflects the Air Quality Index around the Salish Sea, as reported by the Puget Sound Clean Air Agency. But there are other sources for AQI data. PurpleAir manufactures consumer air quality sensors that transmit data to their global network, and they provide a public map of the result. There’s a problem, though: the PurpleAir numbers don’t agree with the Puget Sound Clean Air data. I’ve been uncertain about this in past years, but after a few conversations recently I decided to look into it more carefully.

The AQI is defined differently for different pollutants. We’re mostly looking at PM2.5 for health effects of distant wildfires, and an Air Quality Index of 50 (the top of the “good” category) corresponds to 12 ug/m3 of PM2.5, which is just the mass of stuff bigger than 2.5 microns floating in a cubic meter of air. The AQI is defined by the Environmental Protection Agency. The EPA also defines what are called Federal Reference Methods to measure things like AQI as well as Federal Equivalent Methods, which name products that have been shown to work as well or better than the reference method.

The FRM for PM2.5 pollution is to weigh very precisely a filter that catches nearly everything bigger than 2.5 um, run a constant controlled flow of air through it for 24 hours, then weigh the filter again. The average particle concentration for the flowing air in the last 24 hours is then the mass of new stuff in the filter divided by the flow rate of the air times 24 hours. That’s a simple method that’s pretty easy for anyone with a lab and money to do really precisely. The idea with an FRM is that people will use it to prove to the feds they can consistently build a device that works as well or better than the reference method, the EPA will write a rule in the Federal Register that says the new device is a Federal Equivalent Method for measuring PM2.5 pollution, and then scientists and courts and engineers can trust numbers from manufactured devices without the fed having to build a factory.

EPA and state environmental agencies have other rules, in accordance with the Clean Air Act, that require local governments to operate air pollution monitoring stations. The 2021 revision of the Washington State quality assurance plan for their regulatory air quality monitoring network is available here. Among other things, that plan documents what kinds of instruments regulatory air monitoring stations operate in Washington, and there are a lot of them. Page 53 says each station operates at least two kinds of PM2.5 monitors that are both approved as Federal Equivalents: a continuous monitor, which shines a radiation source (either light or betas) on a detector and estimates very quickly how much stuff gets in the way, and a filter-based monitor that works more like the reference method with a longer sampling period. It’s reasonable to assume the regulatory stations have a good estimate of actual particulate concentrations because every regulatory monitoring station in Washington has an FEM continuous monitor sitting next to a much simpler FEM filter-based monitor and they’re both run by the same trained operator. There’s also one station in the state that operates the state monitors alongside a monitor that works on the same method as the “FEM primary monitor,” which I assume is a federally operated monitor that provides a comparison between the federal and state reports. 

The Puget Sound Clean Air Agency implements the state monitoring policy in this region, so when they say the PurpleAir network data is “uncalibrated” they mean the PA sensors aren’t Federal Equivalents. You could calibrate one by taking it to a regulatory monitoring station and letting it run long enough to figure out what correction is required to make that sensor read the same as a regulatory monitor, but the only agreed-upon way to have confidence in all of them is for PA to go to EPA and get a product approved as an FEM. But the PA monitors are so reliable that PSCAA can apply an automatic correction and report the PA network data alongside the regulatory monitors on its public map (regulatory stations are stars, PA sensors are circles). PSCAA is confident in the PA sensors because people operating precision monitors have been reporting on consumer sensors for a while. This journal article says a set of three PA monitors were more consistent with each other than sets of monitors from other manufacturers, but the PA devices reported values systematically higher than an FEM monitor when PM2.5 density was above 13ug/m3 (which is everything worse than the best AQI category). This report on a national study of PA monitors found they were reliable and that the best estimate of particulate concentration was half the PA value plus a small correction for relative humidity, and discussed a variety of other corrections regulators had considered for PA data.

PS, for some reason I can’t resist adding that most home smoke detectors are continuous monitors that use alphas to detect particulates, so we’ve all got little bits of radioactive metal at home. Totally safe, just don’t eat a lot of them.

I’m interested in getting some experience with WordPress, so I’m building this site using the WordPress Gutenberg editor and then exporting a static site to a GitHub repository. This is the first website I’ve ever made, so it’s been a bit of a learning process.

Here’s what I’ve done so far.

If you’re using this as a tutorial, skip to the end and read the notes first!

Simply Static is the plugin everyone recommends but I couldn’t get it to work. I ran into errors I couldn’t resolve, including a cryptic message that it couldn’t copy some local files when generating the site and a 301 error on the diagnostics page. I was able to figure out that adding quotes around the path to the local directory where the site should be deployed breaks the generator because it doesn’t strip the quotes. I am on Windows though and I have a sneaking suspicion Simply Static would work better on linux, but I was too lazy to pull out another machine to try it out.

I’ve settled on xampp for my local development environment and Staatic to generate the static site. I also tried LocalWP for the local server because it’s very slightly easier to use than xampp, but after an hour playing around with it I couldn’t get Staatic to deploy a site in that environment. Staatic worked right out of the box for me, which was refreshing after my experience with Simply Static.

Set up the development environment

  1. Install xampp. Uncheck the box to open the control panel after the install, because it’ll throw irritating errors when closing the control panel when it runs without admin rights.
  2. Download WordPress. Extract the wordpress directory from the zip file and put it in <xampp install directory>/htdocs/
  3. Open the xampp control panel as administrator, check the boxes for Apache and MySQL in the Service column if they aren’t greyed out, then start them.
  4. Open your browser and go to localhost/phpmyadmin (or click on phpMyAdmin at the top right of the Dashboard page if you got there).
  5. Click on Databases in the toolbar at the top. In the Create database box, make up a name, select Collation at the top of the dropdown menu, and click Create. It doesn’t matter what the name is and we don’t have to configure permissions because we’re going to deploy a static site at the end.

Configure WordPress

  1. Open a browser tab and go to localhost/wordpress. Follow the dialogues to create a new wordpress deployment using the name of the database you just created. The database credentials are username root and a blank password.
  2. Log in at localhost/wordpress with whatever administrator user credentials you just created.
  3. Go to the Plugins tab on the left hand sidebar, check boxes for anything that’s installed and bulk delete them. Click the “add new” button at the top and install Staatic.
  4. Go to the Settings page for Staatic. Select Offline for the Destination URL (you’ll change this when you’re ready to deploy the site).
  5. You have to click Save Changes at the bottom of every settings page.
  6. On the Deployment tab of the Staatic settings page, select Local Directory from the dropdown menu and give it the path to the local directory where you’re syncing the repository for your GitHub site. In the text box for Retain Files/Directories, add lines for .git and anything else you want to live in the repository aside from the website files (like readme.md).
  7. Go to the Publications page under the Staatic tab on the sidebar and click the Publish Now button at the top. If everything is working, you should be able to access a local version of the sample website via index.html in the directory for your GitHub repository.

When you’re ready to deploy the site online, go back to the Staatic plugin Settings page. Under the Build tab, change the Destination URL to / or the URL for your GitHub pages site, publish the site again from the Staatic Publications page, and commit the changes to your repository. There’s a GitHub actions script that has to run after each commit before the page is live, but it should be available in tens of seconds.

A few notes for people who want to try this for themselves

You can’t build a full-featured WordPress site this way. If you aren’t familiar with WordPress, it’s partly designed to make complex products like comments sections and online storefronts easy to deploy. None of that is possible on static sites, which is what GitHub hosts for free.

I’m assuming you’ve set up a GitHub site already. See the official tutorial for instructions.

I don’t know how well WordPress themes work as static sites. I imagine simple ones are fine, but I don’t know if you can do things like frames or animated transitions on static sites. Even if you can, they might not be implemented in WordPress in a way that’s compatible with static site generation. If you’re interested in downloading a third-party theme, you should deploy the theme sample site to GitHub and make sure you can use all the features you want on the live site before investing more time.
