I’ve designed a data model that solves a number of problems databases usually pose for people interested in working with complex and ambiguous data sets. After looking around for similar systems for a while, and conferring with some people around the University of Washington, I think the data model is probably unique and provides some features that aren’t available in any other database. I’m implementing the database with a python package called bibliograph, and I explain the data model in some detail below. The first talk I gave on the database is available here (my talk starts around minute 21).

The problem

Databases are typically considered sources of information, but that isn’t really correct. They’re more like conduits. A database system provides a framework in which someone can make information available in some useful way, but a database always starts out empty. People usually have to put in quite a lot of labor populating a database before anyone can do anything with it. For someone doing long-term research projects with a lot of source material from different places, someone like a historian or a journalist, creating a database means finding ways to extract information from materials that weren’t designed to be represented in a computer, translating that information into a machine-readable format, and feeding it into a piece of software. As anyone with spaces in their surname can attest, it’s very easy to lose important information in that process and wind up with a database that transmits information from external sources very badly.

But there’s an added complication with the materials someone like a historian or a journalist cares about: sources frequently provide conflicting or ambiguous information. From a typical database design perspective that’s a problem because data is supposed to be well-defined and normalizable. If two sources make conflicting statements about the same thing, one statement has to be thrown out or one statement needs to be changed to agree with the other. But somebody like a historian or a journalist isn’t just interested in the information content of a statement. We’re sometimes even more interested in who said something than what they said, so the fact that one of those statements was wrong could be very important. Bibliograph is designed with this in mind – the data model retains all input data verbatim, even if that data is normalized to a different value or cleaned in some way that would destroy the relationship indicated by the initial input statement. The “clean”, normalized relationships are available for queries, but you can always get the messy, ambiguous relationships back again.

The solution

In the bibliograph data model, input statements are interpreted as assertions about string data, and the assertions and strings are abstracted into edges between nodes. An implementation of the model should provide methods to parse input into assertions and strings and then map each string to a single node, many-to-one (so “Alice Smith” and “A. Smith” could get mapped to the same node). Once unique nodes are identified, assertions between strings can be resolved into edges between nodes. Assertions include an additional relation that allows an implementation to keep complete information about the input method, so one can track not only what relationships exist in the input but how the information arrived in the database. The entity relation diagram, above, consists of two basically symmetric parts, a “string” or “input” side and a “node” side. For those familiar with the concept, this can be considered a pair of graph databases, with the string graph consisting of different representations of the node graph.

Links, strings, and nodes

In the bibliograph schema, input statements are stored as “assertions”, which represent links between strings. An assertion is a relationship between an input string, a source string, a target string, a reference string, and a link type. Setting aside the input string for a moment, an assertion can be diagrammed like this

The reference string is interpreted as the context for a directed link between two other strings. So in the statement “Alice said Bob married Charlie,” Alice could be the reference, Bob the source, Charlie the target, and “married” could be the link type.

In my implementation, the input strings are verbatim Python function calls, but they could be anything: file names, archive box numbers, whatever. The purpose is to be able to group assertions together according to the event in which they were inserted into the database. Having each assertion connected to an input operation allows me to efficiently store input metadata. For instance, if I parse a file that has elements that should be interpreted as underbar-delimited lists, I can add an assertion between the input string and the string "_" with link type list_delimiter. All kinds of metadata can be represented this way, so the database can be highly self-descriptive.

Edges are identical to assertions except they refer to rows in the nodes table rather than the strings table, and they don’t include a relation to an input node. So an edge is a relation between a source node, a target node, a reference node, and a link type.

An example use case

I developed bibliograph for my dissertation project, which includes a graph of citations from about 10,000 scientific papers. In addition to citations, I’m interested in tracking people who were acknowledged in the papers and people and institutions who contributed to data sets authors used in their work. To visualize and interpret the graphs, I need clean, normalized relations between people and papers. To understand how the papers fit into a story as historical sources, I need to be able to back out all the details of the context in which graph data originally appeared. The simplest example is personal names – people can have many different correct name representations. Joanne Simpson was an influential meteorologist for instance, and she published under three married names. I can handle this with the following set of assertions

The last three rows come out of a list of aliases I keep up to date as I transcribe information out of papers. When I generate nodes and edges, I have scripts that read alias assertions and map all source and target strings in a set of aliases to the same node. The aliases are ignored when creating edges, so I would normally resolve the assertions above into three edges of link type author. Information about Simpson’s authorship and the papers she wrote could then be retrieved with queries using any of the names in the table above.

This approach means I can preserve relationships between sources outside the database and the information contained in the database. Nodes are abstract objects meant to stand for things in the world like people or places, things that can be represented in many ways. The sources I work with don’t contain people or places, they contain representations of them – the name “Joanne Simpson” is not the same thing as the person called Joanne Simpson. But if I can preserve the original representations used in my sources, I can start asking questions about the sources themselves rather than abstract objects that only live in the database. Did some sources preferentially use certain names? When did Simpson start appearing in print under different names? These are the kinds of questions investigators need to be able to ask, and the answers would be destroyed if Simpson’s name was normalized.

Bibliograph also makes it easier to handle sources from different cultures. For instance a western source might list a Chinese person’s family name in a different order than they would in their own writing, and their name could appear in multiple scripts in different sources. Those situations are all handled with alias assertions (assuming a reasonable text encoding), and afterwards I can always back out which sources used different names for the same person. This also applies to translations that change over time, words that might be translated incorrectly in some sources, or new words introduced to replace old ones. With the bibliograph schema, researchers can start asking questions about what choices people made when creating different sources rather than just identifying sources with similar content.

Finally, working with graphs in the data model is important because I don’t necessarily know ahead of time which relations I’m going to care about. In a graph, relations are stored as (source, target, link type) tuples in a single table rather than keeping different tables for different kinds of relations, so creating a new relation is as simple as adding a new link type. I can pull a lot of citation data from online indexes, but there are still around 100,000 rows of manual entry I have to do (my implementation includes an input system that allows for extremely efficient manual input, but that’s a different story), and I will certainly have new ideas and insights as I work through all those source. This approach allows me to define new relations on the fly in an input file rather than having to change the structure of the database.

Types

An implementation of this data model, or in my implementation a user, must create what’s called a database ontology by defining node types and link types. The bibliograph data model places no restrictions on the ontology. This means the schema can be very precisely tailored to a given application. It can be hard to explain what ontologies are, so I’ll describe one first and then talk about them more generally

In the database for my dissertation, I use these node types:

  • actor: a person or organization
  • work: something an actor made
  • date: a point in time
  • identifier: a marker that should uniquely identify something
  • agreement: a contract between actors
  • acknowledgement: a reference in a work to another actor
  • affiliation: a reference to an actor associated with another actor
  • quote: a piece of a source I transcribed
  • note: a piece of text I want linked to something
  • tag: a piece of otherwise undefined metadata

This is the set of types of things I want to link together in my project. Note that some of them could be subtypes of others – arguably everything on that list except actor qualifies as a subtype of work. But these types aren’t meant to be a logically consistent representation of the world. They were created to aid my specific research project. My approach to data entry requires that I have node types for things like agreements and acknowledgements, but it dictates that other things one might consider nodes are represented by links.

For my dissertation I created these link types:

  • author
  • published
  • volume
  • page
  • doi
  • ads_bibcode
  • title
  • supertitle
  • contains
  • cited
  • affiliated
  • funded
  • acknowledged
  • annotates

Again, this has inconsistencies. Why should volume, page, or title be link types when an affiliation and acknowledgement are node types? Why do I have node types affiliation and acknowledgement and also link types affiliated and acknowledged? These things are dictated exclusively by the way I’ve decided to enter data for my project. I set up a data entry format to compress large amounts of information into single lines for manual entry, and this ontology is determined by that data entry workflow.

So in my case the ontology is designed to efficiently enter data I care about. But there are other approaches to ontologies. The Dublin Core metadata schema and the CIDOC conceptual reference model are gigantic standards meant to provide comprehensive sets of labels for objects and concepts that organizations like libraries and museums can use to create stable database representations of their collections. The bibliograph data model doesn’t restrict the ontology, so an implementation could use the Dublin Core or CIDOC CRM standards, but that wasn’t my concern with this design. The goal here is to connect abstract entities in a database to the data sources used to create them while making as few assumptions as possible about what kind of entities and sources people work with.

Tags

Links sometimes require metadata. For instance, author lists are meaningful in my project and I’d like to know about first and sometimes last authors on papers. Since author is a link type in my setup, I have to attach some kind of metadata to each author link indicating that author’s position in the author list. I do that with many-to-many tables, one connecting assertions to string-valued tags and another connecting edges to nodes associated with the tags.

Strings and nodes don’t need their own many-to-many tag tables. Instead, I define a link type tagged and a node type tag. An assertion can link an arbitrary string to a string representing the tag using the tagged link type. An edge of the same link type would connect the corresponding arbitrary node to the node of type tag corresponding to the tag string.

This is also required for complete documentation of all information in the database. Since edges and assertions can be tagged with arbitrary metadata, they can be largely self-descriptive.

The generalized data model

Note that as I’ve set it up here, the string-side graph doesn’t include a type for the endpoints of the links – they’re all undifferentiated strings. The fully abstract, symmetrical version of this data model would replace the strings by something more general, say “representations” as in the following diagram. In that case representations and nodes would both be instances of link endpoints, called “vertexes” below.

This would be a convenient way to do something like represent an actor with both names and photos. In that case the string value in the representations table could be either a natural name or the name of a file containing the photo. This would allow me to capture the fact that some sources give meaningful information about a person shown in a photo without giving any other identifying information about the person. By using a typed representations table this way, I could more easily connect sources referring to a person by name with other sources that only represented that person with a picture.

I’m interested in getting some experience with WordPress, so I’m building this site using the WordPress Gutenberg editor and then exporting a static site to a GitHub repository. This is the first website I’ve ever made, so it’s been a bit of a learning process.

Here’s what I’ve done so far.

If you’re using this as a tutorial, skip to the end and read the notes first!

Simply Static is the plugin everyone recommends but I couldn’t get it to work. I ran into errors I couldn’t resolve, including a cryptic message that it couldn’t copy some local files when generating the site and a 301 error on the diagnostics page. I was able to figure out that adding quotes around the path to the local directory where the site should be deployed breaks the generator because it doesn’t strip the quotes. I am on Windows though and I have a sneaking suspicion Simply Static would work better on linux, but I was too lazy to pull out another machine to try it out.

I’ve settled on xampp for my local development environment and Staatic to generate the static site. I also tried LocalWP for the local server because it’s very slightly easier to use than xampp, but after an hour playing around with it I couldn’t get Staatic to deploy a site in that environment. Staatic worked right out of the box for me, which was refreshing after my experience with Simply Static.

Set up the development environment

  1. Install xampp. Uncheck the box to open the control panel after the install, because it’ll throw irritating errors when closing the control panel when it runs without admin rights.
  2. Download WordPress. Extract the wordpress directory from the zip file and put it in <xampp install directory>/htdocs/
  3. Open the xampp control panel as administrator, check the boxes for Apache and MySQL in the Service column if they aren’t greyed out, then start them.
  4. Open your browser and go to localhost/phpmyadmin (or click on phpMyAdmin at the top right of the Dashboard page if you got there).
  5. Click on Databases in the toolbar at the top. In the Create database box, make up a name, select Collation at the top of the dropdown menu, and click Create. It doesn’t matter what the name is and we don’t have to configure permissions because we’re going to deploy a static site at the end.

Configure WordPress

  1. Open a browser tab and go to localhost/wordpress. Follow the dialogues to create a new wordpress deployment using the name of the database you just created. The database credentials are username root and a blank password.
  2. Log in at localhost/wordpress with whatever administrator user credentials you just created.
  3. Go to the Plugins tab on the left hand sidebar, check boxes for anything that’s installed and bulk delete them. Click the “add new” button at the top and install Staatic.
  4. Go to the Settings page for Staatic. Select Offline for the Destination URL (you’ll change this when you’re ready to deploy the site).
  5. You have to click Save Changes at the bottom of every settings page.
  6. On the Deployment tab of the Staatic settings page, select Local Directory from the dropdown menu and give it the path to the local directory where you’re syncing the repository for your GitHub site. In the text box for Retain Files/Directories, add lines for .git and anything else you want to live in the repository aside from the website files (like readme.md).
  7. Go to the Publications page under the Staatic tab on the sidebar and click the Publish Now button at the top. If everything is working, you should be able to access a local version of the sample website via index.html in the directory for your GitHub repository.

When you’re ready to deploy the site online, go back to the Staatic plugin Settings page. Under the Build tab, change the Destination URL to / or the URL for your GitHub pages site, publish the site again from the Staatic Publications page, and commit the changes to your repository. There’s a GitHub actions script that has to run after each commit before the page is live, but it should be available in tens of seconds.

A few notes for people who want to try this for themselves

You can’t build a full-featured WordPress site this way. If you aren’t familiar with WordPress, it’s partly designed to make complex products like comments sections and online storefronts easy to deploy. None of that is possible on static sites, which is what GitHub hosts for free.

I’m assuming you’ve set up a GitHub site already. See the official tutorial for instructions.

I don’t know how well WordPress themes work as static sites. I imagine simple ones are fine, but I don’t know if you can do things like frames or animated transitions on static sites. Even if you can, they might not be implemented in WordPress in a way that’s compatible with static site generation. If you’re interested in downloading a third-party theme, you should deploy the theme sample site to GitHub and make sure you can use all the features you want on the live site before investing more time.
